## 先建立三条主线

### 1. C# 不是“带 GC 的 C++”

C# 的语法与 C++ 有不少表面相似处，例如花括号、类、接口、泛型、异常、lambda。但它的核心假设不同：

| 维度 | C++ | C# |
|---|---|---|
| 默认运行模型 | 编译成本机机器码 | 编译为 IL，运行时通常 JIT，也支持 Native AOT |
| 默认内存管理 | 手动、RAII、智能指针 | 托管堆、垃圾回收、`IDisposable` 释放非托管资源 |
| 类型系统 | 值语义很强，指针/引用可深度控制 | 值类型与引用类型明确分离，默认更强调安全 |
| 泛型机制 | 模板，编译期实例化和元编程能力强 | 运行时可感知的泛型，约束系统更明确 |
| 标准库生态 | STL、Boost、平台库、构建系统差异大 | BCL、NuGet、.NET SDK、生态一致性更强 |

### 2. 从“控制一切”切换到“理解运行时协作”

C++ 程序员经常习惯主动控制对象生命周期、内存布局、链接过程和 ABI。C# 依赖 CLR/.NET 运行时提供类型加载、JIT、GC、异常、反射、安全边界和库版本机制。迁移时不要把运行时当成黑箱，也不要试图每一步都按 C++ 方式强行控制。

### 3. C# 工程更像“语言 + SDK + 运行时 + 包生态”的整体

学习 C# 不能只看语法。你需要同时理解：

- `dotnet` CLI：创建、构建、运行、测试、发布。
- `.csproj`：项目元数据、目标框架、包引用、编译选项。
- NuGet：依赖管理。
- .NET 运行时：JIT、GC、程序集加载。
- 框架：ASP.NET Core、EF Core、WPF、WinForms、MAUI、worker service 等。

## 一句话迁移策略

把已有的 C++ 知识当作“概念索引”，不要当作“实现模型”。看到 C# 里的 `class`、`struct`、`using`、`interface`、`generic`、`async`、`delegate` 时，先问两个问题：

1. 这个概念在 C# 运行时下真实做了什么？
2. 这个概念在 C# 工程生态中通常怎么用？
