# 大型工程架构、测试与发布

## 大型 C# 工程的典型分层

常见后端项目结构：

```text
src/
  Product.Api/
  Product.Application/
  Product.Domain/
  Product.Infrastructure/
tests/
  Product.Tests/
```

职责：

- Api：HTTP、认证、请求/响应模型、控制器或 Minimal API。
- Application：用例、命令、查询、事务边界、应用服务。
- Domain：核心业务模型、领域规则、值对象、领域事件。
- Infrastructure：数据库、消息队列、外部服务、文件系统。
- Tests：单元测试、集成测试。

这不是唯一答案，但比把所有代码塞进一个 Web 项目更容易维护。

## 与 C++ 大型工程的差异

| 维度 | C++ | C# |
|---|---|---|
| 模块边界 | 头文件、库、命名空间、链接目标 | 项目、程序集、命名空间、包 |
| 依赖控制 | include/link、CMake target、ABI | ProjectReference、PackageReference、DI |
| 构建产物 | 静态库、动态库、exe | 程序集、NuGet 包、发布目录 |
| 测试接入 | 依项目差异较大 | xUnit/NUnit/MSTest + `dotnet test` |
| 配置 | 编译参数、配置文件、环境变量 | appsettings、Options、环境变量、Secret Manager |

## 依赖方向

一个健康的业务工程通常希望依赖方向稳定：

```text
Api -> Application -> Domain
Infrastructure -> Application/Domain
```

Domain 不应该依赖数据库、HTTP、日志框架、具体外部服务 SDK。这样核心业务能被测试，也能换基础设施实现。

## 依赖注入不是全局变量容器

不好的倾向：

```csharp
public sealed class OrderService
{
    public OrderService(IServiceProvider provider)
    {
    }
}
```

更好的方式：

```csharp
public sealed class OrderService
{
    private readonly IOrderRepository _orders;
    private readonly IClock _clock;

    public OrderService(IOrderRepository orders, IClock clock)
    {
        _orders = orders;
        _clock = clock;
    }
}
```

构造函数参数就是依赖清单。依赖过多往往说明类职责过大。

## 配置

`appsettings.json`：

```json
{
  "Database": {
    "ConnectionString": "..."
  }
}
```

Options 类：

```csharp
public sealed class DatabaseOptions
{
    public required string ConnectionString { get; init; }
}
```

注册：

```csharp
builder.Services.Configure<DatabaseOptions>(
    builder.Configuration.GetSection("Database"));
```

不要把配置散落成到处读取字符串。

## 测试

xUnit 示例：

```csharp
public sealed class PriceCalculatorTests
{
    [Fact]
    public void Calculate_ReturnsDiscountedPrice_ForVipUser()
    {
        var calculator = new PriceCalculator();

        var price = calculator.Calculate(100m, isVip: true);

        Assert.Equal(80m, price);
    }
}
```

运行：

```powershell
dotnet test
```

## 单元测试与集成测试

单元测试：

- 快。
- 不依赖数据库、网络、文件系统。
- 针对纯业务规则。

集成测试：

- 可以启动 WebApplicationFactory。
- 可以连测试数据库或容器。
- 验证真实配置、序列化、路由、ORM 映射。

大型 C# 工程不要只靠 mock，也不要所有测试都打真实数据库。两者要分层。

## 发布

常见发布：

```powershell
dotnet publish -c Release
```

自包含发布：

```powershell
dotnet publish -c Release -r win-x64 --self-contained true
```

单文件发布：

```powershell
dotnet publish -c Release -r win-x64 -p:PublishSingleFile=true
```

Native AOT：

```powershell
dotnet publish -c Release -r win-x64 -p:PublishAot=true
```

发布策略要看场景：服务器、桌面工具、命令行工具、容器、是否允许目标机器安装运行时。

## CI/CD

典型流水线：

```text
restore
build
test
publish
package/containerize
deploy
```

C# 工程的流水线比很多 C++ 工程简单，原因是 SDK、包管理、测试命令更统一。

## 代码质量工具

建议关注：

- nullable reference types。
- analyzers。
- `.editorconfig`。
- format。
- StyleCop 或 Roslyn analyzers。
- 测试覆盖率。
- Dependabot/Renovate 管包版本。

## C++ 到 C# 的工程迁移策略

1. 先画边界，不要逐行翻译。
2. 把纯业务逻辑优先迁移到 C#。
3. 把性能敏感 C++ 留在窄接口后面。
4. 用测试固定行为。
5. 用 NuGet 替代手工第三方库管理。
6. 用 DI 管依赖，不要到处 new 基础设施对象。
7. 用配置系统替代硬编码参数。
8. 发布时明确目标框架、运行时、是否自包含。
