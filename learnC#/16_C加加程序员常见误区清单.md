# C++ 程序员学习 C# 常见误区清单

## 1. 把 C# 当作“语法更简单的 C++”

C# 与 C++ 的相似语法会骗人。真正不同的是运行时、内存管理、项目系统、库生态和工程习惯。

## 2. 以为有 GC 就不用管资源

GC 只管理托管内存。文件、Socket、数据库连接、原生句柄仍然要释放。

正确习惯：

```csharp
using var stream = File.OpenRead(path);
```

## 3. 把 finalizer 当析构函数

C# finalizer 不是 C++ 析构函数。它由 GC 调用，时间不确定，不能承担普通资源管理职责。

## 4. 滥用 `class`

引用类型共享状态很容易造成副作用。小型值对象可以考虑 `readonly record struct` 或不可变 `record`。

## 5. 滥用 `struct`

C# `struct` 是值类型，会复制。大型可变 struct 很容易造成性能和语义问题。

## 6. 忽略 nullable 警告

开启：

```xml
<Nullable>enable</Nullable>
```

不要用 `!` 到处压制警告。可空警告通常是在提醒你的模型边界不清晰。

## 7. 把 `using` 当 `#include`

`using` 导入命名空间，不参与文本包含。依赖关系在 `.csproj` 的包引用、项目引用和程序集引用中。

## 8. 忘记 LINQ 延迟执行

```csharp
var query = users.Where(u => u.IsActive);
```

这通常没有立刻执行。枚举、`ToList`、`Count` 等操作才会触发。

## 9. 多次枚举同一个 `IEnumerable<T>`

可能导致重复计算、重复数据库查询或重复读取流。需要时物化为 `List<T>`。

## 10. 在异步代码里阻塞

避免：

```csharp
var result = GetAsync().Result;
GetAsync().Wait();
Thread.Sleep(1000);
```

优先：

```csharp
var result = await GetAsync();
await Task.Delay(1000);
```

## 11. 以为 `Task` 就是线程

`Task` 表示异步操作，不一定占用独立线程。I/O 异步常常是在等待操作系统完成事件。

## 12. 随手 `Task.Run`

ASP.NET Core 中把业务逻辑包进 `Task.Run` 通常没有好处。CPU 密集工作、后台隔离任务才需要谨慎考虑。

## 13. 忽略 `CancellationToken`

服务端、后台任务、长 I/O 操作都应该传递取消令牌。否则应用关闭、请求断开、超时控制会变差。

## 14. 滥用异常控制普通流程

解析失败、查找不到、用户输入非法通常不该靠异常驱动主流程。Try 模式或结果对象更合适。

## 15. 捕获异常后静默吞掉

```csharp
catch
{
}
```

这会让问题变成不可观测故障。至少记录日志，或转成明确的业务结果。

## 16. 把接口当成仪式

不是每个类都需要一个接口。接口应该表达替换点、边界或测试隔离，而不是机械分层。

## 17. 不理解 DI 生命周期

Singleton 持有 Scoped 服务是常见错误。理解 Singleton、Scoped、Transient 是写 ASP.NET Core 的基础。

## 18. 在属性 getter 里做重活

属性用起来像字段，调用者通常预期它便宜、无副作用。昂贵操作应写成方法。

## 19. 忽略值相等与引用相等

`class` 默认通常是引用相等；`record` 默认是值相等。作为字典键时必须认真处理相等和哈希。

## 20. 对性能问题凭感觉优化

C# 性能问题常来自分配、装箱、闭包、LINQ、多次枚举、日志格式化、锁竞争、数据库查询。先 profiler，再优化。

## 21. 不看 `.csproj`

`.csproj` 决定目标框架、包引用、nullable、隐式 using、unsafe、发布选项。它是工程行为的一部分。

## 22. 只学语法，不做完整项目

建议练习项目：

```text
一个 ASP.NET Core API
  + EF Core 或 Dapper
  + 配置与日志
  + 单元测试
  + 集成测试
  + dotnet publish
```

这比刷语法点更快建立真实 C# 工程感。
