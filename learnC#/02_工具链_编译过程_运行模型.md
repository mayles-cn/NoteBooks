# 工具链、编译过程与运行模型

## C++ 典型流程

C++ 常见流程可以简化为：

```text
.cpp/.h
  -> 预处理
  -> 编译为目标文件 .obj/.o
  -> 链接为可执行文件或动态库
  -> 操作系统加载
  -> 直接执行机器码
```

关键点：

- 头文件参与文本级包含，宏会改变编译输入。
- 模板通常在编译期实例化，可能显著增加编译时间。
- 链接阶段处理符号解析、库依赖、ABI、ODR 等问题。
- 可执行文件通常包含本机机器码。

## C# 典型流程

C# 常见流程可以简化为：

```text
.cs
  -> C# 编译器 Roslyn
  -> IL + 元数据，产出 .dll/.exe 程序集
  -> .NET 运行时加载程序集
  -> JIT 编译为机器码，或提前发布为 Native AOT
  -> 执行
```

关键点：

- 没有 C/C++ 意义上的头文件。
- 类型、方法、属性等信息保存在程序集元数据里。
- 默认由 CLR/.NET 运行时负责加载、验证、JIT、GC、异常处理。
- 可以发布为依赖运行时的应用，也可以发布为自包含应用。
- 对部分场景可以使用 Native AOT 生成本机可执行文件。

## C++ 的“编译链接问题”在 C# 中变成什么

| C++ 问题 | C# 中对应关注点 |
|---|---|
| 头文件包含顺序 | 命名空间、程序集引用、项目引用 |
| 宏污染 | 条件编译符号、Source Generator、MSBuild 属性 |
| ODR 违规 | 程序集版本、类型重复、包版本冲突 |
| ABI 不兼容 | .NET 运行时版本、目标框架、Native interop 边界 |
| 静态库/动态库链接 | NuGet 包、项目引用、运行时依赖 |
| 编译时间过长 | 增量构建、源生成器成本、分析器成本 |

## `.exe` 与 `.dll` 的概念差异

C++ 中：

- `.exe` 通常是可执行文件。
- `.dll` / `.so` / `.dylib` 通常是动态库，暴露二进制 ABI。

C# / .NET 中：

- `.dll` 很多时候是程序集，包含 IL 和元数据。
- 控制台程序也可能主要产出 `.dll`，通过 `dotnet app.dll` 启动。
- Windows 上发布时可能同时有一个 `.exe` 主机文件，用于启动 .NET 程序。
- 程序集边界更像“类型与元数据边界”，不是简单等同 C++ ABI 边界。

## JIT、AOT 与 C++ 编译的对比

| 项目 | C++ | C# 默认 JIT | C# Native AOT |
|---|---|---|---|
| 机器码生成时间 | 构建时 | 运行时按需 | 发布时 |
| 启动速度 | 通常快 | 可能有 JIT 成本 | 通常更快 |
| 峰值优化 | 强依赖编译器与参数 | JIT 可利用运行时信息 | 类似提前编译，但受 AOT 限制 |
| 反射支持 | 非语言核心运行时机制 | 完整支持 | 受限，需要显式配置 |
| 部署体积 | 取决于链接方式 | 依赖运行时或自包含 | 可能更大，但单文件体验好 |

不要把 JIT 简单理解为“慢”。JIT 可以根据当前 CPU、运行时 profile 做优化。也不要把 Native AOT 当作万能开关；它会影响反射、动态加载、泛型和部分库兼容性。

## `dotnet` CLI 常用命令

```powershell
dotnet --info
dotnet new console -n Demo
dotnet build
dotnet run
dotnet test
dotnet publish -c Release
```

迁移初期你应该熟悉这些命令，而不是只依赖 IDE 按钮。

## Debug 与 Release

C++ 中 Debug/Release 常常意味着：

- 优化等级不同。
- 运行库不同。
- 调试信息不同。
- 断言和宏不同。

C# 中也有 Debug/Release，但差异主要来自：

- 编译优化。
- 条件编译符号，如 `DEBUG`。
- 调试符号。
- 发布模式下的 trimming、single-file、ReadyToRun、Native AOT 等选项。

C# 的 Debug 版本不等于“解释执行”。它仍然会编译为 IL，并由运行时执行。

## 你需要形成的新心智模型

从 C++ 迁移时，建议把 C# 程序理解为：

```text
源代码
  -> 项目文件决定如何编译
  -> 程序集承载 IL + 元数据
  -> 运行时提供执行环境
  -> NuGet 与目标框架决定依赖边界
```

不要只盯着 `.cs` 文件。`.csproj`、目标框架、包版本、发布参数同样是程序行为的一部分。
