# 工程管理、项目文件与依赖包

## C++ 工程常见组成

C++ 项目通常包括：

- `.h` / `.hpp`：声明、模板定义、内联函数。
- `.cpp`：实现。
- `CMakeLists.txt`、`.vcxproj`、Makefile、Bazel、Premake 等构建文件。
- 静态库、动态库、系统库、第三方包。
- 编译选项、链接选项、宏定义、包含目录、库目录。

C++ 工程的复杂性很大一部分来自“编译单元 + 链接 + ABI + 平台差异”。

## C# 工程常见组成

C# 项目通常包括：

- `.cs`：源文件。
- `.csproj`：项目文件。
- `.sln`：解决方案文件，用于组织多个项目。
- `Directory.Build.props` / `Directory.Build.targets`：仓库级 MSBuild 配置。
- `global.json`：锁定 .NET SDK 版本。
- `appsettings.json`：应用配置。
- `packages.lock.json`：可选的 NuGet 锁文件。

典型 `.csproj`：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>
</Project>
```

## `.csproj` 与 C++ 构建文件的区别

| 主题 | C++ | C# |
|---|---|---|
| 源文件枚举 | 常需要手工列出或 glob | SDK 风格项目默认包含目录下 `.cs` |
| 头文件 | 关键工程元素 | 没有头文件 |
| 依赖 | include path、lib path、link flags、包管理器 | `PackageReference`、`ProjectReference` |
| 构建系统 | CMake/MSBuild/Make/Bazel 等差异大 | MSBuild + .NET SDK 是主流统一路径 |
| 平台条件 | 宏、生成器表达式、工具链文件 | MSBuild 条件、TargetFramework、RuntimeIdentifier |

## 解决方案 `.sln`

`.sln` 类似“项目集合索引”，常用于：

- 一个 Web API 项目。
- 一个领域层项目。
- 一个基础设施层项目。
- 一个测试项目。
- 一个共享库项目。

例子：

```text
MyApp.sln
  src/MyApp.Api/MyApp.Api.csproj
  src/MyApp.Domain/MyApp.Domain.csproj
  src/MyApp.Infrastructure/MyApp.Infrastructure.csproj
  tests/MyApp.Tests/MyApp.Tests.csproj
```

不要把 `.sln` 理解成 C++ 的最终链接目标。它更像 IDE 和 CLI 共同使用的项目编排文件。

## NuGet 与包引用

C# 依赖通常写在 `.csproj` 中：

```xml
<ItemGroup>
  <PackageReference Include="Serilog.AspNetCore" Version="9.0.0" />
  <PackageReference Include="Dapper" Version="2.1.66" />
</ItemGroup>
```

项目引用通常写成：

```xml
<ItemGroup>
  <ProjectReference Include="..\MyApp.Domain\MyApp.Domain.csproj" />
</ItemGroup>
```

与 C++ 包管理相比，NuGet 的主流体验更统一。你需要关注的不是 include path 和 link path，而是：

- 目标框架是否兼容。
- 包版本是否冲突。
- 是否引入传递依赖。
- 发布时依赖如何复制或裁剪。

## 多目标框架

一个库可以同时面向多个目标框架：

```xml
<PropertyGroup>
  <TargetFrameworks>net8.0;net10.0</TargetFrameworks>
</PropertyGroup>
```

这类似 C++ 同一套源码面向多个平台/编译器构建，但 C# 的维度通常是 .NET 目标框架。

## 常见目录布局

建议从这样的布局开始：

```text
repo/
  src/
    Product.Api/
    Product.Application/
    Product.Domain/
    Product.Infrastructure/
  tests/
    Product.Tests/
  docs/
  Directory.Build.props
  global.json
  Product.sln
```

对于小项目，一个 `.csproj` 足够。不要一开始就过度拆分。

## `using` 不是 C++ 的 `#include`

C++：

```cpp
#include <vector>
```

C#：

```csharp
using System.Collections.Generic;
```

C# 的 `using` 只是导入命名空间，帮助你少写全限定名。它不会把文件文本粘进来，也不会决定链接。真正的依赖边界在项目引用、包引用和程序集引用中。

## 条件编译

C++：

```cpp
#ifdef _WIN32
#endif
```

C#：

```csharp
#if WINDOWS
Console.WriteLine("Windows");
#endif
```

C# 也有预处理指令，但能力比 C++ 宏系统弱得多。一般不要试图用宏模拟 C++ 的模板元编程或条件编译体系。

## 工程迁移建议

如果你从 C++ 项目迁移到 C#，先分清：

- 哪些是业务逻辑，适合用 C# 重写。
- 哪些是性能核心，适合保留 C++。
- 哪些是平台调用，适合封装在窄接口里。
- 哪些构建参数在 C# 中变成 `.csproj` 属性。
- 哪些第三方库有 NuGet 等价物。

迁移的关键不是“逐文件翻译”，而是“按边界重建工程”。
