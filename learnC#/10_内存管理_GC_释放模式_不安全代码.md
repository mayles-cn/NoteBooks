# 内存管理、GC、释放模式与不安全代码

## C++ 的核心内存模型

C++ 程序员通常关注：

- 栈对象生命周期。
- 堆分配与释放。
- RAII。
- 智能指针所有权。
- 移动语义。
- 对象布局、对齐、缓存。
- 悬垂指针、重复释放、未定义行为。

## C# 的核心内存模型

C# 的日常模型不同：

- 引用类型对象通常分配在托管堆。
- 值类型常作为字段、局部变量、数组元素等嵌入存储，具体位置取决于上下文和 JIT。
- GC 负责回收不再可达的托管对象。
- 非托管资源需要显式释放，通常通过 `IDisposable`。
- 运行时可能移动对象，除非 pinning。

## GC 管什么，不管什么

GC 管：

- 托管对象内存。
- 对象可达性。
- 托管堆压缩和代际回收。

GC 不直接管：

- 文件句柄。
- Socket。
- 数据库连接。
- 原生内存。
- GDI、窗口句柄、GPU 资源。
- 其他操作系统资源。

所以“有 GC”不等于“不需要释放资源”。

## `IDisposable`

```csharp
public sealed class ReportWriter : IDisposable
{
    private readonly StreamWriter _writer;

    public ReportWriter(string path)
    {
        _writer = new StreamWriter(path);
    }

    public void Write(string line)
    {
        _writer.WriteLine(line);
    }

    public void Dispose()
    {
        _writer.Dispose();
    }
}
```

使用：

```csharp
using var writer = new ReportWriter("report.txt");
writer.Write("hello");
```

这类似 C++ RAII 的使用体验，但不是同一机制。C++ 在作用域结束时确定调用析构；C# 依赖 `using` 编译成 `try/finally` 调用 `Dispose`。

## `using` 展开后的概念

```csharp
using var stream = File.OpenRead("a.txt");
```

概念上类似：

```csharp
FileStream? stream = null;
try
{
    stream = File.OpenRead("a.txt");
}
finally
{
    stream?.Dispose();
}
```

## Finalizer 谨慎使用

不要把 finalizer 当析构函数。只有持有非托管资源且没有更好封装时才考虑。大多数场景应使用：

- 已有类型的 `Dispose`。
- `SafeHandle`。
- `using`。
- 依赖注入容器生命周期管理。

## 大对象与分配成本

C# 分配引用对象很方便，但不是免费：

- 高频小对象会增加 GC 压力。
- 大对象进入 Large Object Heap。
- 闭包、LINQ、装箱、字符串拼接都可能产生隐性分配。

性能敏感路径要用 profiler，而不是凭感觉改。

## `Span<T>` 与 `Memory<T>`

`Span<T>` 是现代 C# 高性能编程的重要工具：

```csharp
Span<int> buffer = stackalloc int[128];
buffer[0] = 42;
```

特点：

- 表示连续内存片段。
- 可指向数组、栈内存、原生内存等。
- `Span<T>` 是 `ref struct`，不能放到堆上，不能跨 `await`。
- 适合减少复制和分配。

`Memory<T>` 更适合需要跨异步边界保存的内存片段。

## `stackalloc`

```csharp
Span<byte> bytes = stackalloc byte[256];
```

这接近 C++ 栈上临时 buffer 的用法，但要控制大小，避免栈溢出。

## `unsafe` 与指针

C# 可以写指针，但需要 `unsafe`：

```csharp
unsafe
{
    int value = 42;
    int* p = &value;
    Console.WriteLine(*p);
}
```

项目文件需要允许：

```xml
<AllowUnsafeBlocks>true</AllowUnsafeBlocks>
```

只有在互操作、极端性能、底层库场景下才考虑 `unsafe`。普通业务代码不应依赖它。

## Pinning

GC 可能移动对象。如果要把托管对象地址传给原生代码，需要固定：

```csharp
byte[] data = new byte[1024];

unsafe
{
    fixed (byte* p = data)
    {
        // use p
    }
}
```

长期 pinning 会影响 GC 压缩效率，应尽量缩短固定时间。

## C++ RAII 到 C# 的迁移映射

| C++ | C# |
|---|---|
| 栈对象自动析构 | `using` / `try finally` / DI 容器释放 |
| `unique_ptr` | 明确所有者对象 + `IDisposable`，或普通引用由 GC 管 |
| `shared_ptr` | 普通引用 + GC，不表达所有权计数 |
| 析构函数释放资源 | `Dispose` 释放资源，finalizer 兜底且少用 |
| 移动语义 | 通常不需要；大数据用 `Span<T>`、流、池化、不可变模型 |

## 迁移建议

- 托管内存交给 GC，但非托管资源必须释放。
- 使用 `using var` 是 C# 里最接近 RAII 的习惯。
- 性能问题先测量，再考虑 `Span<T>`、对象池、减少装箱、减少闭包。
- 不要为了“像 C++ 一样可控”过早使用 `unsafe`。
