# 异常、资源管理与日志

## C++ 与 C# 异常观念

C++ 项目中异常使用风格差异很大，有些代码库禁用异常。C# 中异常是标准错误处理机制之一，框架大量使用异常表示非预期错误。

但这不代表所有失败都应该抛异常。

## 什么时候抛异常

适合抛异常：

- 参数违反契约。
- 文件不存在但业务要求必须存在。
- 数据库连接失败。
- 外部服务返回不可恢复错误。
- 状态不可能发生但发生了。

不适合抛异常：

- 用户输入验证失败的普通分支。
- 查找不到数据但业务允许。
- 高频控制流。
- 可预期的解析失败。

## 参数检查

```csharp
public void Rename(string name)
{
    ArgumentException.ThrowIfNullOrWhiteSpace(name);
    Name = name;
}
```

常见检查：

```csharp
ArgumentNullException.ThrowIfNull(value);
ArgumentOutOfRangeException.ThrowIfNegative(count);
```

## Try 模式

```csharp
if (int.TryParse(text, out var number))
{
    Console.WriteLine(number);
}
```

适合可预期失败，避免异常成本和噪声。

## 自定义异常

```csharp
public sealed class OrderStateException : Exception
{
    public OrderStateException(string message)
        : base(message)
    {
    }
}
```

不要为每个错误都创建异常类型。只有调用方确实需要区分捕获时，自定义异常才有价值。

## 捕获异常

```csharp
try
{
    service.Process(order);
}
catch (OrderStateException ex)
{
    logger.LogWarning(ex, "Invalid order state: {OrderId}", order.Id);
}
```

避免：

```csharp
catch (Exception)
{
    // 静默吞掉
}
```

吞异常会让系统进入不可观测状态。

## 异常过滤器

```csharp
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    return null;
}
```

异常过滤器比 catch 后再 if 更清晰，也能保留更好的异常处理语义。

## 资源管理

同步释放：

```csharp
using var stream = File.OpenRead(path);
```

异步释放：

```csharp
await using var connection = await OpenConnectionAsync();
```

实现异步释放：

```csharp
public sealed class AsyncResource : IAsyncDisposable
{
    public ValueTask DisposeAsync()
    {
        return ValueTask.CompletedTask;
    }
}
```

## 日志

.NET 常用 `Microsoft.Extensions.Logging` 抽象：

```csharp
public sealed class Worker
{
    private readonly ILogger<Worker> _logger;

    public Worker(ILogger<Worker> logger)
    {
        _logger = logger;
    }

    public void Run(string id)
    {
        _logger.LogInformation("Processing item {ItemId}", id);
    }
}
```

使用结构化日志占位符，不要只拼字符串：

```csharp
logger.LogInformation("User {UserId} logged in", userId);
```

这样日志系统能把 `UserId` 作为字段处理。

## C++ 迁移时的习惯调整

- 不要把错误码风格机械搬过来。C# 框架更习惯异常 + Try 模式并存。
- 不要捕获所有异常后返回默认值。
- 不要在库层随意 `Console.WriteLine`，应使用日志抽象或把错误交给调用方。
- 释放资源依赖 `using`，不是等待 GC。
- 公共方法的参数契约要尽早验证。
